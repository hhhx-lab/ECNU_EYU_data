# G2 results 使用说明

这个目录放的是 G2 在模型正式训练前已经生成好的轻量结果文件。这里不放 NIfTI 影像、不放 nnU-Net 预处理缓存、不放生成模型权重，也不放大规模 synthetic data；这些大文件后面要在训练机器上重新物化或生成。

这些文件主要由 `work_space/G2/code/g2_pretraining_audit.py` 生成。它读取本机外部 BraTS-MET 数据目录，输出 CSV、JSON、Markdown 和少量 DOCX 报告，方便 G1、G2、S1、S2 对齐数据来源、质量控制、训练 split 和后续 synthetic data 接入方式。

## 子目录怎么读

| 子目录 | 内容 | 主要用途 |
|---|---|---|
| `manifests/` | 真实训练集、验证集、corrected labels、GliGAN 兼容 source CSV、nnU-Net 映射表 | 回答“每个病例文件在哪里、能不能用、怎么交给 G1/S1/S2” |
| `stats/` | label 分布、lesion 分布、summary JSON、第一轮 synthetic 分布目标 | 回答“真实数据长什么样，synthetic 应该补什么” |
| `qc/` | synthetic data QC 规则和指标模板 | 回答“G1 生成的样本怎样判定通过、拒绝或人工复查” |
| `splits/` | 固定真实训练/验证 fold | 保证 real-only 与 real+synth 消融使用同一套验证集 |
| `nnunet_raw/` | nnU-Net raw dataset 的轻量占位和 `dataset.json` | 告诉 S1/S2 后续在训练机器上如何物化 nnU-Net 数据集 |
| `reports/` | 给组内沟通和后续实验记录看的总结、模板和报告 | 让人快速理解当前完成了什么、下一步做什么 |

## 推荐使用顺序

1. 先看 `manifests/real_train_manifest.csv`，确认最终可用训练病例、corrected label 覆盖情况和 `final_qc_pass`。
2. 再看 `manifests/g1_gligan_source_cases_v1.csv`，把 G1 生成模型需要的输入路径、bbox、center、lesion 信息交给 G1。
3. 看 `stats/real_label_distribution.csv` 和 `stats/real_lesion_distribution.csv`，决定 synthetic data 要补哪些 label 组合、lesion 大小和多发情况。
4. 用 `qc/qc_rules_v1.md` 和 `qc/qc_metrics_template.csv` 接收 G1 输出，逐例记录 synthetic case 是否能进训练。
5. 用 `splits/splits_final_fold0_realval.json` 固定验证 fold，后面所有 real-only、real+synth、不同 synthetic 比例实验都复用它。
6. 训练机准备好后，S1/S2 按 `manifests/nnunet_case_mapping_realonly.csv` 和 `nnunet_raw/Dataset260_BraTS2026_MET_RealOnly/dataset.json` 物化 nnU-Net 数据。
7. 对外沟通或给队友看进度时，直接看 `reports/G2_训练前数据准备与G1方案对接总结.md` 或同名 DOCX。

## 重要注意事项

1. CSV 里的很多路径是本机绝对路径，换到训练机器后不能直接假设可用。训练机上要么保持相同数据挂载路径，要么根据 `case_id` 重建路径映射。
2. `Validation` 目录中的病例只用于确认数据结构和提交阶段准备，不允许作为 synthetic source，也不进入训练。
3. `final_qc_pass=false` 的病例不能进 G1 source、不能进 nnU-Net real-only 训练，也不能进入固定 split。
4. `g1_gligan_source_cases_v1.csv` 是 GliGAN 兼容格式，但今年可以给 diffusion 使用；关键是沿用它的 source case、bbox、center、label 路径和可复现字段。
5. 这里的 `nnunet_raw/` 只是占位和契约，不代表已经复制好了 NIfTI。
