# manifests 使用说明

这个目录保存“病例级清单”。它们不是模型训练结果，而是 G2 给 G1、S1、S2 使用的数据索引、质控结果和格式转换入口。

## 文件总览

| 文件 | 行数 | 作用 |
|---|---:|---|
| `real_train_manifest_raw.csv` | 1296 | 原始训练病例扫描结果，尚未应用 corrected label 覆盖 |
| `real_train_manifest.csv` | 1296 | 应用 corrected label 后的最终训练 manifest，是 G2 当前真实训练数据的主表 |
| `real_validation_manifest.csv` | 179 | 官方 validation 病例的路径和基础结构检查，不含 seg，不允许作为 synthetic source |
| `corrected_label_overlay.csv` | 2 | 记录哪些病例用官方 corrected label 替换了原始 label |
| `g1_gligan_source_cases_v1.csv` | 1295 | 给 G1/G2 生成模型使用的 GliGAN 兼容 source CSV |
| `nnunet_case_mapping_realonly.csv` | 1295 | 给 S1/S2 在训练机器上物化 real-only nnU-Net 数据集的映射表 |
| `synthetic_generation_manifest_{run_id}.csv` | 每个 run 变化 | G2 从 G1 raw output 自动补建的 synthetic 主清单 |
| `synthetic_candidate_manifest_{run_id}.csv` | 每个 run 变化 | synthetic 候选总表，合并 raw 解析、QC 判定和状态 |
| `synthetic_accepted_manifest_{run_id}.csv` | 每个 run 变化 | 可进入训练或仅进入消融的 synthetic 样本清单 |
| `synthetic_rejected_manifest_{run_id}.csv` | 每个 run 变化 | 被拒绝或需要重新生成的 synthetic 样本清单 |
| `synthetic_normalized_mapping_{run_id}.csv` | 每例 5 行 | raw legacy/native 文件到 2026 标准后缀和 nnU-Net 目标文件名的逐模态映射 |
| `synthetic_generation_manifest_template_g1.csv` | 0 | G1 可直接照着输出、G2 也可用于补建的 manifest 表头 |
| `synthetic_normalized_mapping_template.csv` | 0 | 逐模态标准化映射模板 |

## `real_train_manifest_raw.csv`

这是训练数据的第一轮扫描结果。它逐例记录四个模态和原始 `seg` 是否存在、shape、spacing、affine hash、dtype、原始 label 值域和基础 QC。

主要字段：

| 字段 | 含义 |
|---|---|
| `case_id` | BraTS 病例 ID |
| `split_source` | 病例来自顶层 training 目录还是 `UCSD - Training` 子目录 |
| `case_dir` | 该病例原始目录 |
| `t1n_path`, `t1c_path`, `t2w_path`, `t2f_path` | 四个 MRI 模态的 NIfTI 路径 |
| `raw_seg_path` | 原始分割 label 路径 |
| `has_*` | 对应文件是否存在 |
| `shape_*`, `spacing_*`, `affine_hash_*` | 用来检查模态与 label 是否空间一致 |
| `labels_present` | 原始 label 中出现过的值 |
| `basic_qc_pass`, `basic_qc_reason` | 基础结构 QC 是否通过，以及原因 |

这个文件主要用于追溯，不作为最终训练入口。后续分析优先使用 `real_train_manifest.csv`。

## `real_train_manifest.csv`

这是最终训练 manifest。它在 raw manifest 基础上应用 corrected label 覆盖，并增加最终 QC 字段。

相比 raw 版本新增或更重要的字段：

| 字段 | 含义 |
|---|---|
| `effective_seg_path` | 最终应该使用的 seg 路径，可能是原始 label，也可能是 corrected label |
| `label_source` | `raw` 表示使用原始 label，`corrected` 表示使用 corrected label |
| `has_corrected_label` | 是否找到官方 corrected label |
| `has_illegal_label_after_overlay` | 覆盖后是否仍有非法 label 值 |
| `illegal_label_values_after_overlay` | 覆盖后仍非法的 label 值 |
| `final_qc_pass` | 最终能否进入训练、统计、G1 source 和 split |
| `final_qc_reason` | 最终 QC 原因 |
| `labels_present_after_overlay` | 最终 label 值域 |

目前这个表有 1296 个训练病例，其中 1295 个最终 QC 通过。唯一需要排除的是 `BraTS-MET-01094-002`，原因是 corrected overlay 后仍出现非法 label value `6`。

## `real_validation_manifest.csv`

这是官方 validation 数据清单。它只记录四模态路径、shape、spacing、affine 和基础 QC。validation 没有公开 label，因此不能用于训练、不能用于 G1 synthetic source、不能用于 lesion 统计。

关键字段：

| 字段 | 含义 |
|---|---|
| `case_id` | validation 病例 ID |
| `t1n_path`, `t1c_path`, `t2w_path`, `t2f_path` | 四模态路径 |
| `shape`, `spacing`, `affine_hash` | 空间结构信息 |
| `basic_qc_pass`, `basic_qc_reason` | 基础检查结果 |
| `allowed_as_synthetic_source` | 固定为 `False`，表示绝不作为生成源 |

这个表的用途是提前确认 validation 数据结构，方便后续推理或提交阶段对路径命名做准备。

## `corrected_label_overlay.csv`

这个表只记录需要 corrected label 覆盖的病例。每一行说明原始 seg 和 corrected seg 的路径、值域、shape、spacing、affine 是否匹配，以及是否实际应用。

当前有 2 条记录：

| 病例 | 用途 |
|---|---|
| `BraTS-MET-01094-003` | corrected label 与原始 label 空间匹配，已应用 |
| `BraTS-MET-01184-002` | 原始 label 含非法值 `8`，corrected 后非法值被修正，已应用 |

如果以后官方又提供 corrected labels，要先更新这个表，再重新生成 `real_train_manifest.csv`。

## `g1_gligan_source_cases_v1.csv`

这是给 G1 的核心 source 表。它沿用 2023 GliGAN 风格字段，但今年可以继续给 diffusion 方案使用。

主要用途：

1. 告诉 G1 每个 source case 的四模态路径和最终 label 路径。
2. 提供病灶整体 bbox、center 和 size，方便做 crop、inpainting、条件生成或 ROI 采样。
3. 标记这个病例是否适合直接走 96 crop 的 GliGAN 风格流程。
4. 保留 `allowed_as_synthetic_source` 和 `exclude_reason`，避免把不合适的病例误投给生成模型。

关键字段：

| 字段 | 含义 |
|---|---|
| `id`, `case_id` | source 病例 ID |
| `scan_t1ce`, `scan_t2`, `scan_flair`, `scan_t1` | GliGAN 兼容命名的四模态路径，分别对应 t1c、t2w、t2f、t1n |
| `label` | 最终使用的 label 路径 |
| `center_x`, `center_y`, `center_z` | 病灶整体中心 |
| `x_extreme_*`, `y_extreme_*`, `z_extreme_*` | 病灶整体 bbox 边界 |
| `x_size`, `y_size`, `z_size` | bbox 尺寸 |
| `lesion_component_id` | 当前记录采用的 lesion 组件描述 |
| `label_values` | 该病例出现的 label 值 |
| `usable_for_gligan96` | 是否适合直接用 96 立方 crop 流程 |
| `allowed_as_synthetic_source` | 是否允许作为 synthetic source |
| `exclude_reason` | 不允许或不适合的原因 |

注意：这个表里不是每一行都能无脑生成。G1 使用时必须过滤 `allowed_as_synthetic_source=True`，并根据 `usable_for_gligan96` 决定使用常规 crop 还是单独策略。

## `nnunet_case_mapping_realonly.csv`

这是 S1/S2 用来创建 real-only nnU-Net 训练数据的映射表。它把原始 BraTS ID 映射成 nnU-Net 风格 ID，例如 `BraTS-MET-00001-000` 对应 `BraTSMET_000001`。

关键字段：

| 字段 | 含义 |
|---|---|
| `nnunet_case_id` | 进入 nnU-Net 后使用的病例 ID |
| `source_case_id` | 原始 BraTS 病例 ID |
| `t1n_source_path`, `t1c_source_path`, `t2w_source_path`, `t2f_source_path` | 四模态源路径 |
| `seg_source_path` | 最终 label 源路径，已经考虑 corrected label |
| `label_source` | label 来自 raw 还是 corrected |
| `materialization_status` | 当前状态；`deferred_no_nifti_copy_on_mac` 表示本机未复制 NIfTI，等训练机物化 |

训练机器上要根据这个表创建或软链接：

1. `imagesTr/BraTSMET_000001_0000.nii.gz` 对应 t1n。
2. `imagesTr/BraTSMET_000001_0001.nii.gz` 对应 t1c。
3. `imagesTr/BraTSMET_000001_0002.nii.gz` 对应 t2w。
4. `imagesTr/BraTSMET_000001_0003.nii.gz` 对应 t2f。
5. `labelsTr/BraTSMET_000001.nii.gz` 对应 seg。

这个表不负责 synthetic data。synthetic data 后续应使用单独 manifest，再合并到 real+synth 训练设置里。

## `synthetic_generation_manifest_{run_id}.csv`

这是 G2 接收 G1 raw synthetic run 后自动生成或补齐的主清单。一行对应一个 synthetic case。

它解决四个问题：

1. 这个 synthetic case 从哪个真实 `source_case_id` 来。
2. 它属于哪次 `generation_run_id`、哪个 checkpoint、哪个 seed。
3. G1 输出的是 `legacy_gligan` 后缀、`native_2026` 后缀，还是混用了两套后缀。
4. G2 最终给它分配哪个 `synthetic_final_id` 和 `nnunet_case_id`。

重点字段：

| 字段 | 含义 |
|---|---|
| `synthetic_raw_id` | G1 raw 目录名，保留原始来源 |
| `synthetic_final_id` | G2 分配的标准 synthetic ID，例如 `SYN-MET-000002` |
| `nnunet_case_id` | 未来进入 nnU-Net 时使用的病例 ID，例如 `SYNMET000002` |
| `raw_*_path` | G1 原始输出文件路径，可能是 legacy suffix |
| `normalized_*_path` | G2 计划标准化后的 2026 后缀路径 |
| `nnunet_*_target_path` | 训练机物化 nnU-Net 数据集时的目标路径 |
| `output_suffix_scheme` | `native_2026`、`legacy_gligan`、`mixed` 或 `unknown` |
| `suffix_conversion_action` | G2 对后缀的处理动作 |
| `qc_decision` | `accepted_for_training`、`accepted_for_ablation_only`、`needs_regeneration` 或 `rejected` |

## `synthetic_candidate_manifest_{run_id}.csv`

这是候选总表，保留所有 G1 raw case，不论最后是否通过。它适合用来追责和统计：

1. 多少个 raw case 被接收。
2. 多少个 native、legacy、mixed。
3. 哪些 case 触发了硬拒绝。
4. 哪些 case 只是进入人工复查或 ablation-only。

## `synthetic_accepted_manifest_{run_id}.csv`

这是 G2 放行表。进入这里的样本分两类：

| 字段 | 含义 |
|---|---|
| `accepted_for_training=True` | 可以进入 real+synth 训练候选 |
| `accepted_for_ablation_only=True` | 不建议直接作为主训练数据，但可以用于后缀兼容、legacy 路线、或消融验证 |

注意：accepted 不等于最终比赛策略一定采用。最终是否使用，还要看固定真实验证 fold 上的 real-only vs real+synth 消融。

## `synthetic_rejected_manifest_{run_id}.csv`

这是拒绝表。每一行必须有明确的 `qc_reject_reason` 或 `hard_reject_reason`。

典型拒绝原因：

1. `mixed_suffix_scheme`：同一个 case 同时混用 legacy 和 2026 后缀，容易导致通道错配。
2. `missing_required_file`：缺任一四模态或 seg。
3. `validation_leakage`：source 来自固定 validation fold 或 official validation。
4. `illegal_label_values`：label 不在 `{0,1,2,3,4}`。
5. `shape_inconsistent`、`spacing_inconsistent`、`affine_inconsistent`：几何不一致。

## `synthetic_normalized_mapping_{run_id}.csv`

这是后缀转换和训练机物化的关键映射表。一例 synthetic case 会生成 5 行，分别对应 `t1n/t1c/t2w/t2f/seg`。

它不复制大体积 NIfTI，只记录映射：

| 字段 | 含义 |
|---|---|
| `modality` | 标准模态名，固定为 `t1n/t1c/t2w/t2f/seg` |
| `nnunet_channel` | nnU-Net 通道，`t1n=0000`、`t1c=0001`、`t2w=0002`、`t2f=0003`、`seg=label` |
| `raw_source_path` | G1 原始输出路径，允许是 `scan_t1ce/scan_t1/scan_t2/scan_flair` |
| `normalized_target_path` | G2 标准化后的 2026 后缀路径 |
| `nnunet_target_path` | 训练机上 imagesTr/labelsTr 的目标文件名 |
| `suffix_conversion_action` | `map_legacy_suffix_to_native_2026`、`keep_native_2026_suffix`、`reject_mixed_suffix_scheme` 或 `inspect_unknown_suffix_scheme` |

训练机真正物化数据时，应以这个表为准：accepted 样本复制或软链接到 `nnunet_target_path`，rejected 样本不物化。
