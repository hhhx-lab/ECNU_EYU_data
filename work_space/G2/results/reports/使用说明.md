# reports 使用说明

这个目录保存给人看的报告和后续实验模板。它不作为模型训练输入，主要用于组内沟通、记录当前进展、给 G1/S1/S2 对齐下一步。

## 文件总览

| 文件 | 类型 | 用途 |
|---|---|---|
| `G2_训练前数据准备与G1方案对接总结.md` | 正式总结源稿 | 给 G1、S1、S2 看的当前进展总结 |
| `G2_训练前数据准备与G1方案对接总结.docx` | Word 版总结 | 方便直接发给队友或导师 |
| `g2_pretraining_execution_summary.md` | 执行摘要 | 记录本轮脚本实际生成了什么 |
| `real_data_qc_summary.md` | 真实数据 QC 摘要 | 说明真实训练数据通过、失败、corrected label 情况 |
| `local_data_paths_check.md` | 本机路径检查 | 说明本机外部数据路径是否存在、哪些目录被读取 |
| `G2_synthetic_data_quality_report_template.md` | synthetic QC 报告模板 | G1 交付 synthetic data 后，用它写质量报告 |
| `ablation_plan_template.md` | 消融计划模板 | 后续 real-only、real+synth、不同 synthetic 比例实验用 |
| `README.md` | 简短说明 | 当前目录的极简说明 |

## `G2_训练前数据准备与G1方案对接总结.md` 和 `.docx`

这是最适合发给别人看的总结。内容包括：

1. G2 在模型训练前已经完成了哪些数据准备。
2. 真实训练病例最终可用数量。
3. corrected label 覆盖情况。
4. G1 diffusion/GliGAN 兼容 source 表怎么用。
5. S1/S2 后续如何接 nnU-Net。
6. 哪些工作必须等训练机器或 GPU。

Markdown 是源稿，DOCX 是可编辑交付版。后续如果内容更新，应先改 Markdown，再重新导出 DOCX。

## `g2_pretraining_execution_summary.md`

这是偏执行记录的摘要，说明本轮审计脚本生成了哪些文件、哪些统计结果、哪些任务已经完成。

适合给自己回看，也适合后续提交时写 commit 或周报。

## `real_data_qc_summary.md`

这是对真实训练数据 QC 的简明总结。重点看：

1. 总训练病例数。
2. `final_qc_pass` 数量。
3. 被排除病例和原因。
4. corrected label 如何影响最终 label。

如果要解释为什么某个病例没有进 G1 或 nnU-Net，优先从这里和 `manifests/real_train_manifest.csv` 查。

## `local_data_paths_check.md`

这个文件记录本机数据路径检查结果。它只证明当前 Mac 上这些路径曾经可读，不代表训练机器也有同样路径。

换机器后要重新检查路径，不能只看这个文件就假设训练机数据已就绪。

## `G2_synthetic_data_quality_report_template.md`

这是未来 G1 交付 synthetic data 后使用的报告模板。每批 synthetic data 过 QC 后，建议复制一份并改名，例如：

`G2_synthetic_data_quality_report_g1_smoke_v1.md`

报告里应写清楚：

1. 本批 synthetic data 来自哪个 G1 run。
2. 输入 source manifest 是哪版。
3. 生成数量、通过数量、拒绝数量。
4. 主要拒绝原因。
5. 是否允许进入下一轮 nnU-Net 消融。

## `ablation_plan_template.md`

这是后续实验计划模板。它用来提前定义对比组，避免训练完才临时解释结果。

建议至少包含：

1. real-only baseline。
2. real + smoke accepted synthetic。
3. real + 低比例 synthetic。
4. real + 中比例 synthetic。
5. 不同 generation policy 的对照。

所有对照必须复用 `splits/splits_final_fold0_realval.json` 的真实验证 fold。
