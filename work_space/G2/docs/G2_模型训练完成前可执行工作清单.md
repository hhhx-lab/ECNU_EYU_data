G1 模型训练好之前，G2 不是“没事可做”。相反，G2 此时最应该完成生成数据系统的地基。


5. S1/S2 可以使用 G2 转换出的 real-only nnU-Net dataset 训练 baseline。
6. G1 知道未来必须按什么格式输出 synthetic cases。
7. G2 已经有 QC 规则，能马上筛掉格式错误或医学上明显不合理的 synthetic cases。
8. 后续 real-only vs real+synth 消融只需要替换数据集，不需要重新设计流程。

## 8. 第六步：制定 synthetic target distribution

### 8.1 目的

### 8.2 输出

```text
work_space/G2/data/G2_outputs/stats/target_synthetic_distribution_v1.md
```

### 8.3 文档内容

`target_synthetic_distribution_v1.md` 应包含：

1. 计划生成候选病例数。
2. 计划 accepted 病例数。
3. 每个 source case 最多生成几个 synthetic case。
4. tiny/small/large lesion 目标比例。
5. 多发病灶目标比例。
6. RC 目标比例。
7. ET/SNFH/NETC 组合目标比例。
8. 不允许生成的情况。
9. 第一轮不做的事情。

### 8.4 第一版建议

第一轮建议：

1. 候选 synthetic cases：100-300。
2. accepted synthetic cases：不超过真实训练集的 25%。
3. 每个 source case：最多 1 个 synthetic case。
4. source case：只来自训练集，不来自 validation。
5. 生成模式：local insertion 或 real-label-regeneration。
6. tiny lesion：可以适度提高比例，但不要全部是 tiny。
7. RC：先保守，只基于真实 RC case 做变体。
8. full-case generation：第一版不做。

### 8.5 验收标准

1. G1 知道第一轮应生成什么类型的数据。
2. G2 后续能判断生成结果是否偏离目标分布。
3. S1/S2 能理解 synthetic data 加入训练的目的。

## 10. 第八步：固定真实 train/val/test

### 10.1 目的

所有 synthetic 消融必须使用同一个真实 val/test 划分，避免因为划分变化导致结论不可信。

### 10.2 输出

```text
work_space/G2/results/splits/splits_final_train_val_test.json
work_space/G2/results/splits/splits_final_train_val_test_membership.csv
```

### 10.3 划分原则

1. val/test 只包含真实病例。
2. synthetic case 绝不能出现在 val/test 中。
3. 如果 synthetic case 来自某个 source case，则该 source case 若在 val/test，synthetic 也不能进入主 training。
4. 第一轮固定 829/207/259 三分法。
5. 后续若跑 5 折，必须保证每折都满足同源隔离。

### 10.4 推荐字段

```json
[
  {
    "train": ["BraTSMET_000001", "BraTSMET_000002"],
    "val": ["BraTSMET_000100", "BraTSMET_000101"]
  }
]
```

### 10.5 验收标准

1. S1/S2 使用同一个 split。
2. synthetic 消融版本复用同一 validation。
3. 没有 validation source 派生 synthetic training case。

## 11. 第九步：写 G1-G2 输出契约

### 11.1 目的

在 G1 模型训练完成前，先把输出规范定好，避免模型出来后 G2 无法批量接入。当前契约不能只按理想数据资产来写，因为第一阶段生成模型输入输出实际沿用 2023 GliGAN：CSV 驱动、`96x96x96` ROI、noisy crop + label crop、`real/fake_label` raw case 命名。G2 要做的是把这套 raw 输出接住，再转换成 2026 Task 1 和 nnU-Net 需要的标准数据。

### 11.2 输出

```text
work_space/G2/docs/G1_G2_diffusion_output_contract.md
```

### 11.3 契约必须包含

1. 2023 GliGAN CSV 必填列。
2. 2026 模态到 GliGAN key 的映射：`t1c -> scan_t1ce`、`t1n -> scan_t1`、`t2w -> scan_t2`、`t2f -> scan_flair`。
3. 2026 label 到 region-channel label 的转换策略。
4. `96x96x96` ROI crop 规则。
5. noisy crop + label crop 的 G1 输入 tensor 形状。
6. raw synthetic case 输出目录结构。
7. `real_label` 与 `fake_label` 命名规则。
8. `synthetic_generation_manifest.csv` schema。
9. `generation_log.jsonl` schema。
10. seed 和可复现性。
11. 失败处理。
12. smoke test 标准。
13. G2 QC 标准。
14. G2 到 nnU-Net 的最终导出规范。

### 11.4 G2 对 G1 的最低要求

G1 最少要能交付：

1. 10-20 个 smoke synthetic cases。
2. 每例包含四模态和 `seg`。
3. 每例使用 GliGAN 兼容 raw 命名，或提供 manifest 说明 source case、label kind、label index。
4. 提供 `generation_log.jsonl`。
5. 提供 `synthetic_generation_manifest.csv`，如果 G1 暂时不能生成，至少输出足够信息让 G2 补建。
6. 所有文件能用 nibabel 打开。
7. 所有 label 值域在 `{0,1,2,3,4}`。
8. 同一 seed 能复现。
9. 失败 case 写入 log。

第一版不强制每例包含 `metadata.json`。如果 G1 已经能输出 per-case metadata，可以保留；如果没有，G2 用 manifest/log 承担追踪责任。

### 11.5 验收标准

1. G1 和 G2 对输出结构没有歧义。
2. G2 可以按契约写 QC 脚本。
3. G1 输出 smoke cases 后，G2 不需要手工改名即可接入。

## 18. 常见错误和避免方式

### 18.1 错误：先等 G1 训练好

避免方式：先做 manifest、real-only dataset、split、QC 模板。

### 18.2 错误：直接生成大量 synthetic

避免方式：先做 10-20 例 smoke，再做 100-300 候选。

### 18.3 错误：用 validation 病例派生 synthetic training data

避免方式：manifest 中 `allowed_as_synthetic_source=false`。

### 18.4 错误：把 FID/MS-SSIM 当最终结论

避免方式：最终以固定 internal test 消融为准。

### 18.5 错误：删除小连通域

避免方式：对 MET 小病灶单独评估，不把所有小连通域默认当噪声。

### 18.6 错误：照搬 glioma label mapping

避免方式：以 2026 Task 1 label 为准：1 NETC，2 SNFH，3 ET，4 RC。

### 18.7 错误：写死 shape

避免方式：所有脚本读取 NIfTI header，支持混合 shape/spacing。

### 18.8 错误：没有 manifest/log

避免方式：第一阶段不强制每例 `metadata.json`，但没有 `synthetic_generation_manifest.csv` 或 `generation_log.jsonl` 的 synthetic case 默认不进入训练。

## 20. 最终判断

G1 模型训练完成前，G2 最重要的任务是把“数据能不能被可靠使用”这件事提前解决。

如果这些前置工作做好，G1 给出 synthetic cases 后，G2 可以在很短时间内完成：

```text
接收 synthetic
  -> QC
  -> 转 nnU-Net
  -> 交给 S1/S2
  -> 收集消融结果
  -> 写质量报告
  -> 反馈 G1 下一轮生成方向
```

如果这些前置工作没做，G1 即使训练出模型，G2 也会卡在文件命名、label 修正、数据泄漏、QC 缺失、nnU-Net 转换和验证不可复现上。

所以当前阶段 G2 的最佳策略是：先把真实数据和评估闭环做扎实，再接入 diffusion。生成模型是核心能力，但数据系统决定这项能力能不能真正转化成比赛分数。
