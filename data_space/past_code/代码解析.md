# 往年冠军代码结构深度解析

本文解析本目录下两个历史代码包：

- `往年文章和code/2025`：`BraTS Orchestrator`，把 BraTS 历年靠前算法封装成统一 Python API，再通过 Docker 或 Singularity 调用算法容器。
- `往年文章和code/2023`：`Faking_it team! BraTS submissions`，包含 BraTS 2023 Task 1 冠军、BraTS-ISBI 2024 GoAT 冠军、BraTS 2024 Task 1 冠军，以及若干合成任务/脑膜瘤任务方案。

需要先讲清楚一个关键差别：`2025` 目录不是“训练一个冠军模型”的源码，而是“调度历年冠军/前三名容器”的工具库；`2023` 目录才是更接近比赛复现包的代码，里面既有作者核心脚本，也塞进了 nnUNet、MedNeXt、WDM 这类外部框架源码。

## 总览

整体目录可以分成三块：

```text
往年文章和code/
├── 2025/                       # BraTS Orchestrator，统一调度历史算法容器
│   ├── brats/                  # Python 包源码
│   │   ├── core/               # 推理抽象、Docker/Singularity 后端、任务类
│   │   ├── data/               # 算法元数据 YAML、参数 YAML、额外权重缓存目录
│   │   ├── utils/              # 配置加载、临时目录、日志、Zenodo 下载
│   │   ├── constants.py        # 任务枚举、算法枚举、元数据路径
│   │   └── preprocessing.py    # 调用 brainles_preprocessing 的预处理入口
│   ├── tests/                  # 单元测试
│   ├── docs/                   # Sphinx 文档
│   └── README.md               # 使用说明
│
└── 2023/                       # Faking_it 团队比赛复现包
    ├── Segmentation_Tasks/     # 分割任务：GliGAN 数据增强 + nnUNet/MedNeXt 集成推理
    │   ├── GliGAN/             # 合成肿瘤数据生成器
    │   ├── BraTS2023_inference/
    │   ├── BraTS2024_inference/
    │   ├── BraTS2024_meningioma_inference/
    │   ├── BraTS_ISBI_GoAT_2024_inference/
    │   ├── nnUNet_install/     # 外部 nnUNet v2 源码
    │   ├── mednext/            # 外部 MedNeXt/nnUNet v1 风格源码
    │   └── example/            # dataset.json、plans、splits 示例
    │
    └── Synthesis/              # 合成任务：Task7 缺失模态生成，Task8 inpainting
        ├── Task7/
        └── Task8/
```

如果要从方法角度理解它们，可以概括成两条主线：

- `2025` 主线：输入 MRI 文件 -> 统一改名/临时目录 -> Docker/Singularity 容器推理 -> 输出检查 -> 文件挪回用户指定位置。
- `2023` 主线：用 GliGAN 或 3D WDM 生成更多/更难样本 -> 用 nnUNet、Swin UNETR、MedNeXt 等模型训练 -> 推理时逐模型保存概率图 -> 平均集成 -> 按区域连通域阈值去假阳性 -> 转回 BraTS 标签。

## 2025 BraTS Orchestrator 架构

### 它解决的问题

`2025` 包的目标不是重新实现冠军算法，而是把多个年份、多个任务、多个队伍的算法包装成统一接口。用户只需要选择算法枚举，例如：

```python
from brats import AdultGliomaPreAndPostTreatmentSegmenter
from brats.constants import AdultGliomaPreAndPostTreatmentAlgorithms

segmenter = AdultGliomaPreAndPostTreatmentSegmenter(
    algorithm=AdultGliomaPreAndPostTreatmentAlgorithms.BraTS25_1,
    cuda_devices="0",
)
segmenter.infer_single(
    t1c="case-t1c.nii.gz",
    t1n="case-t1n.nii.gz",
    t2f="case-t2f.nii.gz",
    t2w="case-t2w.nii.gz",
    output_file="segmentation.nii.gz",
)
```

真正的模型权重、网络结构和推理逻辑在 Docker 镜像里，比如 `brainles/brats25_gli_aia-bots:latest`、`brainles/brats23_faking_it:latest` 等。本地 Python 代码只负责“把输入喂给容器”和“把输出拿回来”。

### 分层结构

`2025/brats` 可以拆成六层：

1. 公开 API 层
   `brats/__init__.py` 统一导出 `AdultGliomaPreTreatmentSegmenter`、`MissingMRI`、`Inpainter` 等类，用户通常只接触这一层。

2. 常量与枚举层
   `brats/constants.py` 定义后端类型、任务类型、算法枚举、YAML 文件路径、参数目录、Zenodo 地址、输出命名模板等。

3. 算法配置层
   `brats/data/meta/*.yml` 写清每个算法的作者、年份、名次、论文、Docker 镜像、输入命名规则、是否需要 root、是否支持 CPU、是否需要 Zenodo 额外文件。

4. 抽象推理层
   `brats/core/brats_algorithm.py` 是所有任务的父类，负责单例推理和批量推理的通用流程。

5. 任务适配层
   `segmentation_algorithms.py`、`inpainting_algorithms.py`、`missing_mri_algorithms.py` 负责把不同任务的输入文件改成对应算法需要的 BraTS 文件名。

6. 后端执行层
   `docker.py` 和 `singularity.py` 负责拉镜像、挂载目录、设置 GPU/CPU、运行容器、读取容器日志、检查输出是否存在/为空。

### 单病例推理流程

以四模态分割为例，`infer_single()` 的核心数据流是：

```text
用户传入 t1c/t1n/t2f/t2w
        |
        v
SegmentationAlgorithmWith4Modalities.infer_single()
        |
        v
BraTSAlgorithm._infer_single()
        |
        ├── 创建临时 data_*/output_* 文件夹
        ├── 用 input_name_schema 生成内部 subject_id
        ├── _standardize_single_inputs() 复制并重命名输入
        ├── 选择 Docker 或 Singularity runner
        ├── 运行算法容器
        ├── _sanity_check_output() 检查输出数量和空预测
        └── _process_single_output() 移动到用户指定 output_file
```

这里最重要的是“内部标准命名”。例如成人胶质瘤可能统一成：

```text
BraTS-GLI-00000-100/
├── BraTS-GLI-00000-100-t1c.nii.gz
├── BraTS-GLI-00000-100-t1n.nii.gz
├── BraTS-GLI-00000-100-t2f.nii.gz
└── BraTS-GLI-00000-100-t2w.nii.gz
```

这样不同算法容器都能看到自己熟悉的输入格式。

### 批量推理流程

批量推理要求用户输入目录类似：

```text
input_folder/
├── A/
│   ├── A-t1c.nii.gz
│   ├── A-t1n.nii.gz
│   ├── A-t2f.nii.gz
│   └── A-t2w.nii.gz
└── B/
    ├── B-t1c.nii.gz
    ├── B-t1n.nii.gz
    ├── B-t2f.nii.gz
    └── B-t2w.nii.gz
```

代码会把 `A`、`B` 映射成算法内部名字，例如：

```text
BraTS-GLI-00000-100 -> A
BraTS-GLI-00001-100 -> B
```

容器推理后，再根据这个映射把输出恢复成：

```text
output_folder/
├── A.nii.gz
└── B.nii.gz
```

### Docker 与 Singularity 的区别

Docker 后端有两套挂载规则：

- 2024 及以前算法：使用 MLCube 风格目录 `/mlcube_io0`、`/mlcube_io1`、`/mlcube_io2`、`/mlcube_io3`，运行命令形如 `infer --data_path=/mlcube_io0 --output_path=/mlcube_io2 ...`。
- 2025 算法：使用更直接的 `/input` 和 `/output` 挂载，多数镜像直接运行默认入口。

Singularity 后端复用了 Docker 后端的配置逻辑，但会先把 Docker 镜像转换成 Singularity sandbox，并用 `--bind` 挂载目录。GPU 通过 `--nv` 开启。

### 配置驱动是它的核心设计

以 `adult_glioma_pre_and_post_treatment.yml` 为例，一个算法条目通常包含：

```yaml
BraTS25_1:
  meta:
    authors: Ishika Jain, et al.
    paper: N/A
    challenge: Adult Glioma Segmentation (Pre & Post Treatment)
    rank: 1st
    year: 2025
  run_args:
    docker_image: brainles/brats25_gli_aia-bots:latest
    input_name_schema: BraTS-GLI-{id:05d}-100
    requires_root: true
```

Python 代码通过 `load_algorithms()` 读 YAML，再用 `dacite` 转成 dataclass。新增算法时，理论上只需：

1. 在 `constants.py` 的对应枚举里加一个 key。
2. 在对应 YAML 里加元数据和运行参数。
3. 如果算法有额外权重，补 Zenodo `record_id` 和参数名。

### 2025 代码对我们最有价值的点

对 2026 Task 1 来说，`2025` 目录最值得借鉴的不是模型，而是工程封装方式：

- 把“病例输入格式”和“算法内部格式”隔离开。
- 用临时目录运行推理，避免污染用户数据。
- 用配置文件描述算法，而不是把镜像名、参数、论文信息写死在代码里。
- 同时支持单病例和批量推理。
- 对输出做 sanity check，至少确认数量够、预测不是全 0。
- 把 Docker/Singularity 视为后端，可切换。

它不适合作为训练代码参考，因为容器内部算法不可见；它适合作为后续我们封装推理/提交系统的模板。

## 2023 Faking_it 复现包架构

### 它解决的问题

`2023` 目录是一个更完整的比赛复现包。它的核心思想可以概括成：

```text
真实数据有限
    |
    v
生成式模型制造更多肿瘤形状和影像
    |
    v
训练多个分割网络，让不同模型互补
    |
    v
推理时集成多个 checkpoint 的概率图
    |
    v
用连通域体积阈值清理小假阳性
    |
    v
输出符合 BraTS 标签约定的 nii.gz
```

这里的“Just faking it”主要指 synthetic data augmentation：不是只在真实样本上做旋转、翻转，而是用生成模型制造新的肿瘤标签、把肿瘤贴到脑内合理位置、再用模态生成器重建局部影像。

### 顶层任务

README 里列了这些任务：

- BraTS 2023 Task 1：Adult Glioma Segmentation，冠军。
- BraTS-ISBI 2024 GoAT：Generalizability Across Tumors，冠军。
- BraTS 2024 Task 1：Adult Glioma Post Treatment，冠军。
- BraTS 2024 Task 3：Meningioma Radiotherapy，第三名。
- BraTS 2024 Task 7：Synthesis Global，缺失 MRI 模态生成。
- BraTS 2024 Task 8：Synthesis Local，inpainting。

所以 `2023` 包不是单一任务项目，而是一个“多任务复现仓库”。

### 分割任务主架构

分割任务主要由三部分组成：

```text
Segmentation_Tasks/
├── GliGAN/                         # 合成数据生成
├── *_inference/                    # 不同任务的提交推理脚本
├── nnUNet_install/                 # nnUNet v2 外部源码
├── mednext/                        # MedNeXt 外部源码
└── example/                        # nnUNet 数据配置示例
```

#### 第一层：GliGAN 合成数据

GliGAN 里有两类生成器：

1. 标签生成器
   输入随机向量，输出一个 3D 肿瘤标签块。它基于 alpha-GAN 风格结构，包含 Generator、Encoder、Discriminator、Code Discriminator。

2. 模态生成器
   对每个 MRI 模态分别训练一个局部重建模型。输入是“被高斯噪声遮住的局部 patch + 肿瘤标签”，输出是“看起来真实的带肿瘤局部影像”。

合成过程大致是：

```text
选择一个真实病例
    |
    v
从 label generator 或真实标签库拿一个肿瘤形状
    |
    v
随机找一个脑内可放置位置
    |
    v
把新标签贴进原标签，形成新 segmentation
    |
    v
对每个模态，取 96x96x96 局部 patch
    |
    v
用模态生成器把噪声区域重建成带肿瘤影像
    |
    v
保存新病例，加入 nnUNet 训练集
```

关键细节：

- CSV 文件记录每个肿瘤的中心、边界框、大小，后续训练和粘贴都依赖这些坐标。
- 只使用三轴都不超过 `96` 的肿瘤作为 GliGAN 训练样本，因为生成器 patch 固定为 `96x96x96`。
- `gaussian_tumour` 只在肿瘤区加噪声；`gaussian_extended` 会在肿瘤周围球形区域也加噪声，让模型学习更自然的边界过渡。

#### 第二层：nnUNet/MedNeXt 分割训练

作者没有从零实现分割网络，而是站在 nnUNet 生态上：

- 标准 nnUNet 3D full resolution。
- Swin UNETR，作为 transformer/attention 分支。
- 2021 winner 风格的大 nnUNet 变体。
- BraTS 2024 后处理胶质瘤任务还引入 MedNeXt。

它们互补的原因是：

- nnUNet 对医学分割很稳，局部纹理和尺度适配强。
- Swin UNETR 有全局注意力，对较复杂空间关系有帮助。
- 大模型/MedNeXt 提供不同归纳偏置，集成能降低单模型失误。

训练数据分成多种实验编号：

- `G`：真实数据 + 用真实标签驱动 GliGAN 生成的数据。
- `rG`：真实数据 + 随机标签生成器产生标签，再由 GliGAN 生成的数据。
- `R`：注册增强生成的数据。
- `B`：标准 nnUNet。
- `S`：Swin UNETR。
- `L`：Large nnUNet / 2021 winner 风格。
- `M`：MedNeXt。

例如 `rGB_rGL_rGS` 表示使用随机标签 GliGAN 数据训练出来的三种网络：nnUNet、Large nnUNet、Swin UNETR。

#### 第三层：提交推理脚本

每个任务都有一个 `main.py` 和一个 `infer_low_disk*.py`：

```text
main.py
    负责设置环境变量、选择 ensemble_code、设置阈值、串起五步流程

infer_low_disk*.py
    负责具体的数据转换、逐模型推理、概率图累加、阈值后处理、标签转回
```

五步流程基本固定：

1. `convert_data_step()`
   把 BraTS 文件名转换成 nnUNet 的 `_0000/_0001/_0002/_0003` 输入格式。

2. `perform_inference_step()`
   根据 `ensemble_code` 生成一串 `nnUNetv2_predict` 或 MedNeXt 推理命令，逐个模型跑。

3. `ensemble_predictions()`
   读取每个模型保存的 `.npz` 概率图，除以模型数量后累加。这个做法比保存所有预测再统一 ensemble 更省磁盘。

4. `thresholding_step()`
   先把预测转成 WT/TC/ET/RC/GTV 等区域，再用连通域体积阈值删除小结构，减少 lesion-wise 指标下的假阳性惩罚。

5. `convert_back_BraTS_step()`
   把 nnUNet 内部连续标签映射回 BraTS 官方标签，并复制到最终输出目录。

### 标签与区域映射

BraTS 原始标签和训练/后处理区域不是一回事。代码经常先把标签转成“区域通道”，最后再转回官方标签。

BraTS 2023 成人胶质瘤：

| 区域 | 含义 | 由原始标签组成 |
| --- | --- | --- |
| TC | Tumor Core | label 1 或 label 3 |
| WT | Whole Tumor | label 1、2、3 |
| ET | Enhancing Tumor | label 3 |

BraTS 2024 Post-treatment Glioma：

| 区域 | 含义 | 由原始标签组成 |
| --- | --- | --- |
| TC | Tumor Core | label 1 或 label 3 |
| WT | Whole Tumor | label 1、2、3 |
| ET | Enhancing Tumor | label 3 |
| RC | Resection Cavity | label 4 |

BraTS 2024 Meningioma RT：

| 区域 | 含义 |
| --- | --- |
| GTV | Gross Tumor Volume，label 1 |

这种区域化设计很重要，因为 BraTS 评价不是只看单个 label，而是看临床定义的组合区域。

### 后处理为什么重要

从 2023 开始，BraTS 更强调 lesion-wise Dice 和 lesion-wise HD95。小假阳性以前可能只影响一点总体 Dice，现在会被按病灶级别重罚。冠军代码因此大量使用体积阈值：

- BraTS 2023 / GoAT 常见默认：WT `250`、TC `150`、ET `100`。
- BraTS 2024 Post-treatment Glioma 默认：WT `50`、TC `0`、ET `0`、RC `50`。
- Meningioma RT 默认：GTV `0`。

后处理逻辑是：

```text
预测标签
    |
    v
转成区域二值图 WT/TC/ET/RC/GTV
    |
    v
3D 连通域分析
    |
    v
删除体积小于阈值的连通块
    |
    v
按优先级合回 label 图
```

这一步对比赛指标非常现实：它不改变模型结构，但能显著减少小噪声预测造成的假阳性。

## 合成任务 Task7/Task8 架构

### Task7：Missing MRI

Task7 目标是给定 4 个 MRI 模态中的 3 个，生成缺失的那一个。代码使用 3D Wavelet Diffusion Model。

核心思想：

```text
四模态 MRI
    |
    v
3D Haar DWT，小波分解成 8 个频带
    |
    v
把已知模态和待生成模态指示通道拼接
    |
    v
扩散模型在小波空间采样
    |
    v
IDWT 还原成 3D NIfTI
```

作者尝试了几种训练模式：

- `default`：默认 3D WDM，所有模态都可能参与噪声。
- `known_all_time`：已知模态训练/推理时尽量保持已知。
- `known_3_to_gen_1`：推荐模式，只生成缺失模态；输入是 `4 模态 * 8 小波通道 + 4 个模态指示通道 = 36`，输出是 `1 模态 * 8 小波通道 = 8`。

Task7 的推理脚本会自动检查病例目录缺了哪个模态，然后只生成这个缺失模态。

### Task8：Inpainting

Task8 目标是给定 `t1n-voided` 和 `mask`，把被遮掉的区域补成健康组织。

核心思想：

```text
T1n voided + mask
    |
    v
mask 指示需要生成的 ROI
    |
    v
条件 3D WDM 只在 ROI 学习/采样
    |
    v
生成完整 T1n
    |
    v
用非 ROI 原始强度做线性校正，减少亮度不一致
```

Task8 多了一个数据生成器 `data_creator`，用于制造更多“健康 mask + voided T1n”训练样本：

1. 从真实病例取原 T1n 和病灶 mask。
2. 用标签生成器或真实健康 mask 随机生成更多 ROI。
3. 在脑内找到不重叠且不越界的位置。
4. 把对应区域置零，形成新的 voided volume。
5. 保存新数据和 metadata。

### WDM 代码与 GliGAN 的关系

GliGAN 主要服务分割任务：制造“有肿瘤”的训练样本。

WDM 主要服务合成任务：

- Task7 生成缺失模态。
- Task8 修补局部缺失区域。

两者都属于生成式增强/生成式提交，但目标不同：

| 模块 | 生成对象 | 用途 |
| --- | --- | --- |
| GliGAN label generator | 肿瘤标签块 | 制造新病灶形状 |
| GliGAN modality generator | 带肿瘤局部 MRI patch | 增强分割训练集 |
| Task7 WDM | 缺失 MRI 模态 | 直接提交合成结果 |
| Task8 WDM | mask 内健康组织 | 直接提交 inpainting 结果 |

## 2023 代码对我们最有价值的点

对 2026 BraTS-MET 任务，最值得继承的是这些策略：

1. 统一输入转换
   先把官方数据转换成训练/推理框架的标准格式，最后再转换回来。这能减少路径、命名、标签约定混乱。

2. 概率图级 ensemble
   不要只投票最终 label。保存并平均概率图，通常比多数投票更稳定。

3. 低磁盘集成
   每跑完一个模型就把概率图除以总模型数并累加，再删掉该模型的临时输出。大规模 3D 数据很需要这种思路。

4. 区域级连通域阈值
   MET 任务也会遇到小病灶、假阳性、病灶级评价问题，阈值不能照抄，但“按区域/按病灶后处理”很值得复用。

5. 生成式数据增强
   脑转移瘤样本可能存在类别不平衡、小病灶难学、扫描来源差异。GliGAN 的思想可以启发我们做病灶级合成增强，但直接复用代码需要重写标签定义和数据路径。

6. 多模型互补
   冠军方案不是押宝一个网络，而是让 nnUNet、Swin UNETR、MedNeXt/大模型互补。2026 也可以设计轻量版 ensemble。

## 主要风险和坑

### 2025 包的坑

- Docker 镜像外部依赖强，容器内源码不可见。
- 很多算法默认 GPU；`force_cpu=True` 只有 YAML 标记 `cpu_compatible: true` 的算法可用。
- 2024 及以前和 2025 算法的容器接口不一样，改后端时要小心。
- 输出 sanity check 只检查数量和全 0，不代表医学质量可靠。

### 2023 包的坑

- 代码里有大量绝对路径或旧集群路径，需要本地化。
- nnUNet/MedNeXt 是嵌入的第三方源码，改动成本高。
- 推理脚本大量依赖硬编码 dataset id 和 trainer 名字，例如 `Dataset232_BraTS_2023_rGANs`。
- 合成数据代码默认读取/缓存大量 3D 数据，内存压力大。
- 一些脚本中导入了未显式导入的模块，例如 `pickle` 在部分 helper 中使用时依赖上下文，移植时要整理。
- label 映射强依赖任务，不可直接套到 BraTS-MET。
